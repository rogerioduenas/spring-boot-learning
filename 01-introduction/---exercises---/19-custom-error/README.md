# 🏋️ Exercise 19 — Customizing Error Responses with `DefaultErrorAttributes`

## 📝 Description

Globally customize the default JSON error responses generated by Spring Boot in order to remove unnecessary fields (such as `trace` or `path`) and provide a cleaner, more concise error format.

---

## ⚙️ Technical Objectives

- Learn how to use and customize Spring's `DefaultErrorAttributes` component.
- Understand how Spring Boot generates default error responses.
- Modify the global error structure for all application errors.

---

## 📋 Business Rules

1. Create a class that extends or overrides the default `DefaultErrorAttributes` Bean.
2. Customize the method responsible for generating the error attributes.
3. Retrieve the default error attribute map generated by Spring.
4. Remove unnecessary keys from the response.
5. Keep only the following fields:
    - `timestamp`
    - `status`
    - `error`
6. Ensure that the customization affects all error responses globally.

---

## 🎯 Expected Error Format

Whenever an error occurs, the response should contain only the allowed fields.

Example:

```json
{
  "timestamp": "2026-06-08T12:30:45.123Z",
  "status": 404,
  "error": "Not Found"
}
```

Fields commonly included by Spring Boot, such as the following, should no longer appear:

```json
{
  "path": "/invalid-route",
  "trace": "...",
  "message": "...",
  "requestId": "..."
}
```

## ✅ Success Criteria

The exercise is considered complete when:

- A custom `DefaultErrorAttributes` implementation is configured.
- Spring Boot error responses are modified globally.
- Unwanted fields are removed from every error response.
- Only the fields `timestamp`, `status`, and `error` remain visible in the JSON output.
