# 🏋️ Exercise 26 — Post-Execution Error Handling with AOP (`@AfterThrowing`)

## 📝 Description

Create a system that notifies administrators (via console logs) whenever a critical failure or unexpected system error occurs in the business layer using AOP.

## ⚙️ Technical Objectives

- Practice using `@AfterThrowing` to capture exceptions before they reach the application's global Exception Handler.

## 📂 Required Structure

Class:

- `PaymentService` (Create a method that intentionally throws a generic exception such as `IllegalStateException`.)

Aspect Class:

- `ErrorLoggerAspect`

## 🎯 Routes and HTTP Methods

### POST `/payments/process`

## 📋 Functional Requirements

- The Service method must fail intentionally by throwing an exception.
- The `@AfterThrowing` Aspect must capture the error and print a red alert message to the console containing the exception message.
- After that, the `@RestControllerAdvice` should handle the exception normally and return **409 Conflict** to the client.

## 🧪 Postman Test Cases

Trigger the endpoint.

Postman should receive a **409 Conflict** response generated by the Advice, while your IDE terminal should display the alert log generated by the Aspect.
